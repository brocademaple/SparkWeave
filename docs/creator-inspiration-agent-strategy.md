# 创作者素材金字塔与多 Agent 编排策略

## 一句话定位

SparkWeave 面向创作者的核心价值，是把路上、闲暇时、突然冒出的灵感保存成可长期复用的素材弹药库，并在用户真正想创作时，通过 RAG、Prompt 模板和多 Agent 编排，帮用户跨过从 0 到 1 最痛苦的起步阶段。

用户负责捕捉灵光一闪。SparkWeave 负责在需要时检索、理解、组合、编排和输出。

在 Obsidian companion 的方向下，Obsidian 可以继续作为长期知识库和素材库；SparkWeave 承担个性化创作孵化 Agent 的角色。

## 核心用户场景

很多创作者并不缺灵感。他们缺的是把灵感持续推导下去所需的执行力、判断力和耐力。

典型场景：

- 写小说的人突然想到一个人名、一个人设、一个场景、一个冲突，但这些碎片还不足以构成一部长篇。
- 科技评论作者持续记录 AI 产品评测、数字员工案例、协同办公观察，但真正写公众号时还要重新挑选、归纳和立意。
- 专业领域创作者积累了行业事实、客户洞察、方法论片段和案例证据，但整理成报告、课程或咨询简报时成本很高。

SparkWeave 不要求用户每次灵感出现时都把它变成完整作品。它先保护素材，等创作目标出现时，再把素材组合成作品起点。

## 素材金字塔模型

### 第一级：收件箱整理类型

第一级是用户收件箱整理时看到的素材类型。它不应该被写死为全局枚举，而应该按创作者档案提供预设，并允许用户增删、改名和排序。

系统可以提供默认选项，例如：

- 项目材料
- 文章观点
- 待验证假设
- 今日行动

但这些只应作为初始模板。不同创作者对素材的理解方式差异很大，所以类型需要可配置。

### 第二级：末端素材灵感片段

第二级是每个素材类型下的具体片段。它们可以很小，甚至只是一个词、一个名字、一句台词、一条观察、一段产品体验。

末端素材的价值不在于当下完整，而在于未来可以被检索、组合和复用。

### 输出层：作品起点

当用户提出创作目标时，系统从末端素材中检索相关片段，再通过 Agent 编排生成作品起点，例如：

- 小说题材建议
- 故事核
- 人物关系
- 章节大纲
- 公众号文章结构
- 专业报告框架
- 咨询简报
- 课程提纲

## 按创作者档案配置素材类型

### 文学创作者

默认一级类型：

- 人名
- 人设
- 情节片段
- 场景
- 台词
- 世界观
- 冲突
- 氛围
- 主题母题
- 结局灵感

常见末端素材：

```text
人名：沈归雪
人设：一个不相信神明的祭司，却能听见死者的梦。
场景：暴雨夜，废弃戏台上只有一盏红灯还亮着。
冲突：她必须救一个仇人的孩子，才能保住自己家族的秘密。
氛围：旧城、雾、铜铃、潮湿木头、压抑的礼法。
```

可生成输出：

- 题材判断：古风、现代言情、校园、未来架空、末世等。
- 人物组合：主角、对手、关键配角。
- 故事核：主角想要什么、阻碍是什么、代价是什么。
- 大纲起点：三幕结构或前 10 章方向。

### 科技创作者

默认一级类型：

- 产品评测
- 案例观察
- 观点判断
- 趋势线索
- 反例
- 标题灵感
- 用户反馈
- 商业模式
- 竞品差异
- 待验证问题

常见末端素材：

```text
产品评测：某数字员工产品的 onboarding 很强，但实际任务拆解能力弱。
案例观察：协同办公产品正在从文档协作转向流程代理。
观点判断：AI 工具的价值不在生成内容，而在减少跨系统执行摩擦。
反例：很多 AI 办公产品演示惊艳，但真实组织落地时缺权限和流程治理。
标题灵感：为什么数字员工不是下一个 Notion？
```

可生成输出：

- 公众号推文结构
- 标题候选
- 核心观点
- 案例证据表
- 产品横评
- 趋势评论
- 选题库

### 专业领域创作者

默认一级类型：

- 行业事实
- 客户洞察
- 方法论片段
- 案例证据
- 待验证假设
- 交付结构
- 风险提示
- 专家观点
- 数据引用
- 复盘结论

常见末端素材：

```text
行业事实：中小企业愿意买工具，但很难持续维护流程。
客户洞察：客户说想要自动化，其实更需要有人帮他明确工作边界。
方法论片段：先识别重复决策，再谈流程自动化。
案例证据：一个团队上线工具后，真正提升来自每周固定复盘。
待验证假设：AI Agent 的引入顺序应该先从低风险、低权限任务开始。
```

可生成输出：

- 行业分析框架
- 咨询简报
- 客户方案初稿
- 课程提纲
- 方法论文档
- 调研问题清单

## 多 Agent 编排方式

MVP 采用“按目标启动”的编排方式。用户提出一个明确目标后，系统再启动 Agent 流水线。

示例目标：

```text
帮我基于已有素材想一部长篇小说。
```

```text
整合我之前对数字员工和协同办公产品的评测，写一篇公众号推文。
```

后台可以做轻量整理，例如自动补标签、识别素材类型、记录可能主题。但正式输出不应在用户无目标时自动生成大篇内容。

### 1. 需求分析 Agent

职责：

- 解析用户一句话或一件事。
- 判断用户真正想要的输出类型。
- 明确体裁、受众、长度、风格、约束和质量标准。
- 决定需要调用哪些素材类型和模板。

输入：

```text
我想根据之前攒的那些人名和场景，看看能不能发展成长篇小说。
```

输出：

```text
创作目标：长篇小说起点
优先素材：人名、人设、场景、冲突、世界观、氛围
候选题材：古风、未来架空、悬疑奇幻
输出格式：题材判断 + 故事核 + 人物关系 + 前 10 章大纲
质量标准：不要生成完整正文，先判断哪个方向最有潜力
```

### 2. 素材抓取 Agent

职责：

- 从末端节点中检索素材。
- 结合标签、语义相似度、时间、用户偏好和历史采用记录排序。
- 返回可引用的素材片段，而不直接生成作品。

检索信号：

- 用户自定义素材类型
- 标签
- 关键词
- 语义向量
- 来源笔记
- 最近使用时间
- 用户收藏或采用记录
- 被拒绝或忽略记录

输出应包含引用依据：

```text
选中素材：
1. 沈归雪：人名，曾被标记为“古风/宿命感”
2. 不信神明的祭司：人设，和“宗教秩序/秘密”高度相关
3. 废弃戏台红灯：场景，氛围强，适合作为开篇
4. 救仇人的孩子：冲突，具备道德困境
```

### 3. 编排 Agent

职责：

- 将素材从散点整理成逻辑结构。
- 对小说素材，关注人物动机、冲突、世界观、情节推进。
- 对评论文章，关注观点、证据、反例、结论。
- 对专业内容，关注问题定义、案例、方法论和交付结构。

编排不是润色。它的重点是结构和判断。

小说编排示例：

```text
结构判断：
- “沈归雪”适合做主角，因为名字和祭司设定气质一致。
- “不信神明的祭司”提供身份悖论。
- “救仇人的孩子”提供第一卷核心冲突。
- “废弃戏台红灯”适合作为开篇场景。

题材建议：
古风悬疑奇幻最合适，其次是未来架空。
```

科技评论编排示例：

```text
结构判断：
- 数字员工 onboarding 强但任务拆解弱，是现象。
- 协同办公从文档协作转向流程代理，是趋势。
- 真实落地缺权限和流程治理，是反例。

核心观点：
数字员工的难点不在会不会生成，而在能不能进入组织流程。
```

### 4. 润色输出 Agent

职责：

- 根据用户偏好和模板输出成品草稿或创作包。
- 保留素材引用，方便用户回溯。
- 避免把早期灵感过度定稿。
- 允许用户选择“简报版、详细版、可直接发布版”。

输出类型：

- 小说创作包
- 公众号文章方案
- 专业报告框架
- 选题建议
- 项目 brief
- 下一步行动清单

## RAG、Prompt 和模板的职责分工

### RAG

RAG 负责从用户素材库中找到真正相关的片段。

它解决的问题：

- 用户不需要自己翻素材。
- 系统可以引用真实来源。
- 输出建立在用户过去的灵感上，而不是泛泛生成。

### Prompt

Prompt 负责让 Agent 按稳定角色工作。

它解决的问题：

- 需求分析 Agent 不直接写稿。
- 素材抓取 Agent 不擅自编故事。
- 编排 Agent 先做结构判断。
- 润色输出 Agent 才进入表达加工。

### 模板

模板负责承接不同创作者的输出格式。

它解决的问题：

- 文学创作者需要故事核、人物关系和章节方向。
- 科技创作者需要观点、证据、反例和公众号结构。
- 专业创作者需要问题定义、方法论、案例和交付框架。

## MVP 示例一：小说创作

### 输入素材

```text
1. 人名：沈归雪
2. 人设：一个不相信神明的祭司，却能听见死者的梦。
3. 场景：暴雨夜，废弃戏台上只有一盏红灯还亮着。
4. 台词：“你拜的不是神，是你不敢承认的罪。”
5. 冲突：她必须救一个仇人的孩子，才能保住自己家族的秘密。
6. 世界观：城中每个人出生时都会得到一枚铜铃，铃响代表命运改写。
7. 氛围：旧城、雾、铜铃、潮湿木头、压抑的礼法。
8. 结局灵感：主角最后亲手敲碎自己的铜铃。
```

### 用户目标

```text
帮我看看这些素材适合写成什么题材的长篇小说，并给我一个起步方案。
```

### Agent 输出

```text
题材判断：
最适合古风悬疑奇幻。素材中的祭司、铜铃、旧城、戏台和宿命感高度一致。

故事核：
一个不信神的祭司能听见死者的梦。她为了保住家族秘密，被迫救下仇人的孩子，却因此发现整座城的命运都被铜铃制度操控。

人物组合：
- 主角：沈归雪，不信神的祭司。
- 关键孩子：仇人之子，也是铜铃制度的漏洞。
- 对手：维护礼法和神谕秩序的人。

核心冲突：
主角想摆脱神权秩序，但她赖以生存的身份正来自这个秩序。

前 10 章方向：
1. 暴雨夜戏台开篇，主角听见死者梦境。
2. 仇人孩子出现，铜铃异常响起。
3. 主角被迫救人。
4. 家族秘密第一次露出裂缝。
5. 城中铜铃制度的规则被展示。
6. 孩子说出与死者梦境相同的话。
7. 主角被神庙审问。
8. 废弃戏台再次出现，红灯成为线索。
9. 主角发现铜铃可以被伪造。
10. 第一卷目标确立：查清谁在改写命运。
```

## MVP 示例二：科技评论

### 输入素材

```text
1. 数字员工产品 onboarding 很强，但真实任务拆解能力弱。
2. 协同办公产品正在从文档协作转向流程代理。
3. AI 工具的价值不在生成内容，而在减少跨系统执行摩擦。
4. 很多 AI 办公产品演示惊艳，但组织落地时缺权限和流程治理。
5. 用户愿意尝试 AI 助手，但不愿把关键业务权限直接交出去。
6. 标题灵感：为什么数字员工不是下一个 Notion？
```

### 用户目标

```text
整合我之前对数字员工和协同办公产品的观察，写一篇公众号推文方案。
```

### Agent 输出

```text
标题候选：
1. 为什么数字员工不是下一个 Notion？
2. 数字员工的真正难点，不是生成，而是进入流程
3. AI 办公产品演示很强，落地为什么很难？

核心观点：
数字员工的价值不在于像人一样生成内容，而在于能否被安全地嵌入组织流程，减少跨系统执行摩擦。

文章结构：
1. 开头：数字员工演示越来越像“万能同事”，但真实使用中经常卡在任务拆解和权限边界。
2. 现象：onboarding 强、演示强、落地弱。
3. 趋势：协同办公正在从文档协作转向流程代理。
4. 关键问题：组织不缺 AI 生成能力，缺的是权限、流程治理和责任边界。
5. 反例：用户愿意试用助手，但不愿直接交出关键业务权限。
6. 结论：真正的机会在低风险、低权限、可回溯的流程节点。

开头草稿：
过去一年，很多数字员工产品都在努力证明一件事：AI 可以像同事一样工作。它能写文档、建任务、总结会议、推动流程。但真实进入组织之后，问题很快暴露出来：演示里的数字员工很聪明，工作流里的数字员工却常常无处落脚。
```

## MVP 示例三：内容创作者：为什么收藏越多，产出越少？

这个案例用于验证 SparkWeave 是否真的能把“保存素材”推进到“生成可开始的创作方向”。

很多内容创作者并不缺资料，真正困难的是收藏太多之后反而更难开始。收藏按钮会带来一种虚假的进步感：用户把文章、截图、金句、案例和链接存进 Obsidian 或其他素材库时，会感觉自己已经完成了学习和准备。但真正进入写作时，这些素材仍然是松散的、未排序的、没有主题权重的。

真正的问题在于想法没有被组织。素材库保存了“我看过什么”，但没有回答“这些东西能共同说明什么”。一个好工具要帮助人发现主题、聚合证据、判断哪些片段已经足以支撑一个选题。

在 Obsidian companion 方向下，Obsidian 继续承担长期素材库职责；SparkWeave 承担创作孵化和 Agent 编排层职责。它读取用户已有素材，识别其中反复出现的问题、情绪、案例和观点，再把它们编织成可验证的选题。

### 输入素材

```text
1. 观点：收藏越多，我越觉得自己在进步，但真正写东西时还是空白。
2. 观察：很多知识管理工具强化的是保存动作，不是产出动作。
3. 案例：Obsidian 里有 300 条 AI 写作笔记，但只有 2 篇真正发布。
4. 反例：有时一条粗糙的随手记录，比十篇收藏文章更容易变成选题。
5. 用户反馈：我不是不知道写什么，是不知道这些东西之间有什么关系。
6. 标题灵感：为什么收藏越多，产出越少？
```

### 用户目标

```text
帮我把“为什么收藏越多，产出越少？”发展成一个可写的内容选题，并用我已有收藏和笔记做支撑。
```

### Agent 输出

```text
问题定义：
收藏让用户获得“已经开始”的心理反馈，但素材没有经过主题归并、证据排序和观点判断，所以进入创作时仍然像从零开始。

核心观点：
收藏本身不是创作进度。创作进度来自主题被发现、素材被组织、观点被验证。

文章结构：
1. 开头：收藏夹越来越满，发布列表却没有变长。
2. 虚假进步感：保存动作提供即时奖励，让用户误以为已经完成了思考。
3. 真正卡点：想法之间没有结构，素材没有形成主题、证据链和表达顺序。
4. 工具误区：很多工具只优化“存得更快”，没有优化“怎么开始写”。
5. SparkWeave 机会：把 Obsidian 里的素材按主题聚类，生成选题、提纲和可引用片段。
6. 结论：好工具应该帮人发现主题，而不只是保存资料。

MVP 功能验证点：
- 能否从 5-10 条收藏/笔记中发现 1-3 个可写主题。
- 能否解释每个主题引用了哪些原始素材。
- 能否生成一份用户愿意继续编辑的文章起点。
```

## 新手样本

以下三个样本用于 MVP 首轮测试。它们都遵循同一个边界：Obsidian 可以作为素材库，SparkWeave 是创作孵化和 Agent 编排层。

### 样本 A：内容创作者

适合用户：

- 经常收藏文章、截图、金句和案例，但发布频率不稳定的公众号、播客、视频或小红书创作者。
- 已经有 Obsidian、微信收藏、Readwise 或本地 Markdown 素材，但缺少把素材变成选题的启动器。

初始输入：

```text
目标：把最近收藏的知识管理和 AI 写作素材整理成一个可发布选题。
素材：
1. 收藏越多越焦虑，像是在还没开始前就背上了债。
2. 很多工具都在强化保存动作。
3. 真正能发布的内容，通常来自几个素材之间的冲突。
4. Obsidian 里有很多标签，但标签不等于主题。
5. 标题灵感：为什么收藏越多，产出越少？
```

SparkWeave 编织结果：

```text
发现主题：
1. 收藏带来的虚假进步感。
2. 素材没有被组织，所以无法进入创作。
3. 好工具应该帮人发现主题，而不只是保存资料。

输出：
- 公众号文章标题候选。
- 文章结构。
- 每一节引用的素材片段。
- 下一步写作动作：先写开头 300 字，再补 2 个真实案例。
```

验证指标：

- 用户能在 3 分钟内选中一个想继续写的主题。
- 输出至少引用 3 条原始素材。
- 用户反馈为“可用”或“部分可用”。
- 用户愿意把结果写回 Obsidian 作为下一篇文章草稿。

### 样本 B：产品/创业者

适合用户：

- 正在打磨产品机会、MVP 假设、用户访谈问题或 pitch 结构的产品经理、独立开发者和早期创业者。
- 平时积累了用户反馈、竞品观察、功能灵感和商业模式片段，但缺少把它们压缩成验证路径的工具。

初始输入：

```text
目标：判断“个人素材 Agent”是否值得做成 MVP。
素材：
1. 创作者愿意保存素材，但很少回头整理。
2. 用户说想要 AI 帮写作，但真实卡点是启动和组织。
3. Obsidian 用户不想迁移知识库。
4. 竞品大多强调 capture、search 或 chat。
5. 假设：如果能从已有素材生成可用选题，用户会更愿意每天捕捉。
```

SparkWeave 编织结果：

```text
机会判断：
产品切入点是站在 Obsidian 之上提供创作孵化和 Agent 编排。

MVP 假设：
当用户输入 5-10 条素材后，SparkWeave 能生成一个足够清晰的选题、提纲和下一步行动。

验证方案：
1. 找 5 位内容创作者导入一组真实素材。
2. 让系统生成 3 个候选主题。
3. 记录用户是否选择其中一个继续编辑。
4. 访谈用户为什么采用或拒绝。
```

验证指标：

- 5 位测试用户中至少 3 位选择继续编辑某个输出。
- 每次输出能形成明确的“问题-观点-证据-下一步”结构。
- 用户能说清 SparkWeave 与 Obsidian 的分工。
- 至少 2 位用户愿意第二天继续添加素材。

### 样本 C：个人复盘者

适合用户：

- 有日记、周记、晨间笔记或 Obsidian Daily Notes 习惯，但复盘时容易只停留在记录层的人。
- 想从生活、工作和学习碎片中识别长期主题、重复模式和下一步行动的人。

初始输入：

```text
目标：把这一周的零散记录整理成一次复盘。
素材：
1. 周一：会议很多，真正推进的事情很少。
2. 周二：晚上写了 40 分钟，比白天效率高。
3. 周三：看到一个产品案例，想到自己也可以做小范围验证。
4. 周四：反复拖延同一个任务，因为下一步不清楚。
5. 周五：如果提前写好问题清单，会议会短很多。
```

SparkWeave 编织结果：

```text
复盘主题：
本周的核心问题集中在任务启动前：缺少明确问题和下一步。

模式识别：
- 高价值工作集中在低打扰时段。
- 会议消耗来自准备不足和边界不清。
- 拖延通常出现在任务没有被拆成下一步动作时。

下周行动：
1. 每天上午只保留一个最小推进目标。
2. 会议前写 3 个问题，会议后只保留一个行动项。
3. 把晚上 40 分钟设置成固定创作窗口。
```

验证指标：

- 用户能在 10 分钟内完成一次周复盘。
- 输出至少识别 1 个重复模式和 1 个可执行行动。
- 用户愿意把复盘结果写回 Obsidian Daily Notes 或 Weekly Note。
- 下周复盘时，用户能标记上一周行动是否被执行。

## 与 Obsidian 的关系

Obsidian 适合作为素材长期存放地：

- Markdown 原文
- 双链关系
- 标签和属性
- Canvas
- 本地可控的知识库

SparkWeave 适合作为创作孵化层：

- 识别素材类型
- 检索末端灵感
- 编排多条素材
- 生成作品起点
- 学习用户偏好
- 把结果写回 Obsidian

用户不需要迁移出 Obsidian。SparkWeave 只在“我要开始创作了”这个时刻提供加速。

## 产品原则

1. 素材类型必须允许用户自定义。
2. 系统预设只作为起点，不作为固定边界。
3. 捕捉时尽量轻，编排时再深入。
4. Agent 各司其职，避免一个 Prompt 包办所有事情。
5. 输出必须引用使用了哪些素材。
6. MVP 先生成起点，不追求完整终稿。
7. 用户反馈会影响未来检索和编排。
8. ADHD 用户尤其需要低摩擦捕捉和低压力启动。

## MVP 成功标准

第一次可用的 MVP 应该让用户完成以下闭环：

1. 选择创作者档案。
2. 使用系统预设素材类型，也可以改名或新增。
3. 捕捉 5-10 条末端灵感。
4. 提出一个创作目标。
5. 系统检索素材并说明选择依据。
6. 系统生成一个作品起点。
7. 用户反馈“可用、部分可用、不可用”。

如果用户感受到“我只负责保存灵感，系统能帮我把它们推成一个可开始的方向”，这个 MVP 就达到了验证目标。
